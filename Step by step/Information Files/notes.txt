Title: DBB-ECC: Random Double Bit and Burst Error Correction Code for HBM3

================================================================================
1. PROBLEM STATEMENT
================================================================================
- DRAM technology scaling causes reliability issues (higher bit error rates)
- HBM3 uses on-die ECC with Single Symbol Correction (SSC) RS codes
- SSC can only correct errors within a single 16-bit symbol
- Cannot address randomly scattered errors (errors in multiple symbols)
- Need robust ECC that handles BOTH burst errors AND scattered errors

================================================================================
2. EXISTING SOLUTIONS & LIMITATIONS
================================================================================
- Hamming codes: Low redundancy but cannot correct symbol errors
- BCH DEC: Cannot correct single symbol errors (SSE)
- RS16 (19,17): Corrects burst errors but not scattered errors
- DEC-SbEC [13]: Combines RS + BCH, but requires 50% more parity bits
- Unity ECC [14]: Reduces overhead but has large decoder complexity
  (large look-up table, impractical for on-die implementation)

Key Challenge:
- Inherent tradeoff between error correction capability and hardware cost
- As error correction capability increases, so does hardware overhead
  (parity bits and encoding/decoding complexity)
- Existing solutions [13], [14] designed for 8-bit symbol size (DIMMs)
- HBM3 necessitates 16-bit symbol size due to long data length
- Difference in symbol size further increases hardware cost

================================================================================
3. DBB-ECC PROPOSED SOLUTION
================================================================================
Key Innovation:
- Corrects BOTH single 16-bit symbol errors AND random double bit errors
- Uses same parity bits as standard SSC RS codes (no storage overhead)
- Exploits syndrome patterns of double bit errors to reduce decoder complexity

Code Used: RS16 (19,17)
- 16-bit symbol size
- 17 data symbols (272-bit data)
- 2 parity symbols (32-bit parity)
- Redundancy rate: 11.76%

================================================================================
3.1 REED-SOLOMON CODE FUNDAMENTALS
================================================================================
RS codes are a subclass of nonbinary BCH codes constructed with symbols
from a Galois Field GF(q), where q is typically a power of two.
- For q = 2^m, m-bit symbols are used (m = symbol size)
- Symbol error: SBE or multiple bit errors belonging to the same symbol
- RS codes correct t symbol errors with 2t parity symbols
- Concatenated k data symbols + 2t parity symbols = n codeword symbols

H-Matrix (Parity Check Matrix):
The H-matrix dictates the structure of the encoder/decoder.
For SSC RS (n, k) code:

        [ 1      1      ...    1        1      1    0 ]
    H = [                                              ]
        [ T_0    T_1    ...    T_{k-2}  T_{k-1} 0    1 ]

Where T_0, T_1, ..., T_{k-1} are different elements in GF(2^m).
When constructing H-matrix, k elements are selected from GF(2^m)
such that each syndrome of target errors is unique.

Syndrome Computation:
- Syndromes calculated by multiplying 2×n parity check matrix with n codeword symbols
- Used for error detection and correction
- When codeword has no error, syndromes = zeros
- If any syndrome is nonzero, error is DETECTED

For Single Symbol Error (SSE) at location i with error pattern E_i:
    [ S_0 ]   [   E_i   ]
    [     ] = [         ]
    [ S_1 ]   [ T_i·E_i ]

Where S_0 and S_1 are m-bit syndrome symbols, E_i is 16-bit error symbol.

Key Relationship for SSE:
    T_i · S_0 ⊕ S_1 = 0    (Equation 3)

Where ⊕ is bitwise XOR operation.

RS decoder computes T_j·S_0 ⊕ S_1 for 0 ≤ j ≤ k-1.
Result is zero only when j = i (error location found).
Using error location i and error pattern S_0, SSE is corrected.

On-Die ECC in HBM3:
- RS16 (19,17) code employed [4]
- 32-bit parity protects 272-bit data from single 16-bit symbol error
- Combined with system CRC in memory controller achieves >99.99% detection rate

================================================================================
4. SYNDROME PATTERNS OF RANDOM DOUBLE BIT ERRORS
================================================================================
Three location types for random 2-bit errors:

Type 1: Two SBEs in two different DATA symbols (D_i0, D_i1)
        - 136 possible locations (17C2)
        - S0 = E_i0 XOR E_i1

Type 2: One SBE in DATA symbol, one in PARITY symbol P0
        - 17 possible locations (17C1)
        - S0 = E_i0 XOR E_i1

Type 3: One SBE in DATA symbol, one in PARITY symbol P1
        - 17 possible locations (17C1)
        - S0 = E_i0 (error pattern in data symbol)

Key Observation:
- Count of "1"s in S0 symbol can only be 0, 1, or 2 for random 2-bit errors
- This pattern is exploited for efficient decoding

================================================================================
5. DBB-ECC DECODING METHOD (DETAILED EXPLANATION)
================================================================================

THE SETUP:
- Data: 272 bits split into 17 data symbols (each 16 bits)
- Parity: 32 bits (2 parity symbols: P0 and P1)
- Syndromes: S0 and S1 (16-bit each) - computed from received data

THE CORE PROBLEM:
When two random single-bit errors occur in DIFFERENT symbols, standard RS
codes can't fix them. DBB-ECC solves this by cleverly analyzing syndrome S0.

--------------------------------------------------------------------------------
KEY INSIGHT: Counting "1"s in S0
--------------------------------------------------------------------------------
When a random 2-bit error occurs, the number of "1"s in S0 can ONLY be 0, 1,
or 2. This tells us which type of error happened:

  Count of 1s in S0 | What it means
  ------------------|----------------------------------------------
        2           | Two errors in different positions (different bit patterns)
        1           | One error in data, one in parity P1
        0           | Two errors with identical bit patterns

--------------------------------------------------------------------------------
THE THREE CASES:
--------------------------------------------------------------------------------

CASE A: S0 has TWO "1"s (Type 1 and Type 2 candidates, E_j0 ≠ E_j1)
  Example: S0 = 0000000000000101
           This means one error flipped bit 0, another flipped bit 2

  - Split S0 into two single-bit error patterns: E_j0 = bit0, E_j1 = bit2
  - Starting from LSB, if bit position in S0 is 1, first error pattern E_j0
    is estimated as SBE symbol for that position
  - Second error pattern E_j1 = S0 ⊕ E_j0
  - Two SBE symbols can be switched, so 2 pairs of (E_j0, E_j1) considered

  For Type 1 (both errors in data symbols D_i0, D_i1):
      T_i0·E_i0 ⊕ T_i1·E_i1 ⊕ S_1 = 0    (Equation 4)

  For Type 2 (one error in data D_i0, one in parity P0):
      T_i0·E_i0 ⊕ S_1 = 0                 (Equation 5)

  Sublocator 1 computes these for 0 ≤ j0 < j1 ≤ 16 using estimated patterns.

CASE B: S0 has ONE "1" (Type 3 only)
  Example: S0 = 0000000000001000
           This means error in data symbol is at bit 3

  - Error symbol E_j0 in data symbol D_j0 is estimated as S0 symbol
  - Error symbol E_j1 in parity P1 is estimated as one of 16 SBE symbols
  - 16 pairs of (E_j0, E_j1) are considered

  For Type 3 (one error in data D_i0, one in parity P1):
      T_i0·E_i0 ⊕ E_i1 ⊕ S_1 = 0         (Equation 6)

  Sublocator 2 computes T_j0·E_j0 ⊕ E_j1 ⊕ S_1 for 0 ≤ j0 < j1 ≤ 16.

CASE C: S0 is ZERO (Type 1 and Type 2 candidates, E_j0 = E_j1)
  Example: S0 = 0000000000000000
           This means BOTH errors flipped the SAME bit position

  - Error patterns E_j0 and E_j1 estimated as same SBE symbols
  - Zero S0 provides NO information about error location
  - Must compare computed S1 with stored syndromes in syndrome table
  - Only 2,448 cases out of 43,776 total (5.6%) - small table needed!
  
  Sublocator 3 compares S1 symbol with precomputed syndromes.

--------------------------------------------------------------------------------
WHY THIS IS CLEVER:
--------------------------------------------------------------------------------
1. NO EXTRA PARITY BITS - Uses same 32-bit parity as standard RS codes
2. SMALL LOOKUP TABLE - Only 5.6% of cases need table lookup
3. FAST - Most cases (94.4%) solved by simple math, not table search
4. PARALLEL PROCESSING - SSE and DBE locators run simultaneously

6. DECODER ARCHITECTURE
================================================================================
Components:
1. Syndrome Generator - computes syndromes from codeword
2. SSE Locator - standard RS decoding for single symbol errors
3. DBE Locator - novel double bit error correction
   - Syndrome Splitter: estimates error patterns from S0
   - Sublocator 1: handles count=2 cases
   - Sublocator 2: handles count=1 cases
   - Sublocator 3: handles count=0 cases (syndrome table)
   - DBE Info Generator: determines result based on S0 count
4. Decision Logic - selects valid error info
5. Error Corrector - XOR operations to correct errors

--------------------------------------------------------------------------------
SYNDROME SPLITTER HARDWARE:
--------------------------------------------------------------------------------
Purpose: Estimates error patterns (E_j0, E_j1) from S0 symbol as SBE symbols

Algorithm:
1. Start from LSB of S0
2. Check each bit of S0 to determine first error pattern E_j0
3. If bit position in S0 is "1", E_j0 = SBE symbol for that position
4. Once E_j0 derived, compute E_j1 = S0 ⊕ E_j0 (bitwise XOR)
5. Send estimated patterns to Sublocator 1

--------------------------------------------------------------------------------
SUBLOCATOR OPERATIONS:
--------------------------------------------------------------------------------

Sublocator 1 (Count = 2):
- Computes: T_j0·E_j0 ⊕ T_j1·E_j1 ⊕ S1  for Type 1
- Computes: T_j0·E_j0 ⊕ S1              for Type 2
- Range: 0 ≤ j0 < j1 ≤ 16
- Uses estimated patterns (T_j0, T_j1) from syndrome splitter
- When result = 0, error location is determined

Sublocator 2 (Count = 1):
- Computes: T_j0·E_j0 ⊕ E_j1 ⊕ S1       for Type 3
- Range: 0 ≤ j0 < j1 ≤ 16
- Uses S0 symbol and SBE symbol as estimated error patterns
- When result = 0, error location is determined

Sublocator 3 (Count = 0):
- Compares S1 symbol with precomputed syndromes in syndrome table
- Syndrome table stores 2,448 cases (5.6% of all 43,776 DBE cases)
- Look-up table accounts for < 10% of DBB-ECC decoder area

--------------------------------------------------------------------------------
ERROR INFORMATION FORMAT:
--------------------------------------------------------------------------------
- Error location: 17-bit one-hot vector (dimensions = number of data symbols)
- SSE error pattern: 16-bit
- DBE error pattern: 32-bit (two 16-bit patterns)

Decision Logic:
- Selects valid error information from SSE locator and DBE locator
- Forwards selected info to error corrector

Error Corrector:
- Performs XOR operations between erroneous data symbols and error patterns
- Outputs corrected data

--------------------------------------------------------------------------------
- SBE (Single Bit Error): Single bit flip anywhere in codeword

- SSE (Single Symbol Error): Multiple bit errors within one 16-bit symbol
- DASE (Double Adjacent Symbol Error): Errors spanning two adjacent symbols
  (reported as most frequently observed errors [4])

7.4 DETAILED ANALYSIS
--------------------------------------------------------------------------------

HBM2E [15]:
- Uses SBE correction with DBE detection for both on-die and system ECC
- Lowest parity bits used
- Can correct SBE and most DBE (99.6%)
- CANNOT correct SSE and DASE (only detects them)

HBM3 [4]:
- Symbol-based ECC to address burst errors
- Corrects SBE, SSE, and DBE within single 16-bit symbol (0.6%)
- Most DBE (98.8%) only detected, not corrected
- Some DBE (0.6%) remain UNDETECTED (SDC risk)
- Some DASE (0.6%) remain UNDETECTED (SDC risk)

DBB-ECC (Proposed):
- Perfectly corrects SBE, SSE, and DBE (100%)
- NO storage overhead vs HBM3
- Slightly lower undetected DASE (0.4% vs 0.6%)
- Overall improved reliability against common error scenarios

--------------------------------------------------------------------------------
7.5 SILENT DATA CORRUPTION (SDC) ANALYSIS
--------------------------------------------------------------------------------
SDC occurs when ECC mis-corrects an uncorrectable error, resulting in
undetected errors being falsely reported as corrected.

SDC Risk Comparison:
- HBM2E: 0% SDC for all tested scenarios
- HBM3:  0.6% SDC for DBE, 0.6% SDC for DASE
- DBB-ECC: 0% SDC for SBE/DBE/SSE, 0.4% SDC for DASE

DBB-ECC eliminates SDC risk for the most common error types (SBE, DBE, SSE).

12.2 GALOIS FIELD MULTIPLIER (gf_mul_16.v)
--------------------------------------------------------------------------------

PRIMITIVE POLYNOMIAL: x^16 + x^5 + x^3 + x + 1 (0x1002B)
  - Lower 16 bits used for reduction: POLY = 0x002B

MODULES IMPLEMENTED:

1. gf_mul_16 (Sequential Multiplier with Registered Output)
   - Algorithm: Shift-and-add (Russian peasant multiplication)
   - Latency: 1 cycle (combinational logic + output register)
   - Interface: clk, rst_n, valid_in, a[15:0], b[15:0] → valid_out, p[15:0]
   - Use case: Default multiplier for encoder/decoder

2. gf_mul_16_pipelined (4-Stage Pipelined Multiplier) - NEW
   - Algorithm: 4-stage pipeline, each stage processes 4 bits
   - Latency: 4 cycles initial, 1 result/cycle throughput
   - Interface: clk, rst_n, valid_in, a[15:0], b[15:0] → valid_out, p[15:0]
   - Use case: High-frequency timing-critical paths

3. gf_mul_param (Parameterized Registered Multiplier) - NEW
   - Algorithm: Generative shift-and-add with configurable symbol size
   - Parameters: M (symbol size: 8, 16, or 32), POLY (reduction polynomial)
   - Supported polynomials:
     * M=8:  x^8 + x^4 + x^3 + x^2 + 1 (POLY = 0x1D)
     * M=16: x^16 + x^5 + x^3 + x + 1 (POLY = 0x002B)
     * M=32: x^32 + x^7 + x^5 + x^3 + x^2 + x + 1 (POLY = 0x000000AF)
   - Latency: 1 cycle
   - Use case: Reusable for different ECC configurations

4. gf_mul_param_comb (Parameterized Combinational Multiplier) - NEW
   - Algorithm: Pure combinational generative shift-and-add
   - Parameters: M (symbol size), POLY (reduction polynomial)
   - Latency: Combinational (0 cycles)
   - Use case: Internal use in inverse computation, low-latency paths

5. gf_mul_16_formal_props (Formal Verification Properties) - NEW

12.3 GALOIS FIELD INVERSE (gf_inv_16.v)
--------------------------------------------------------------------------------

PRIMITIVE POLYNOMIAL: x^16 + x^5 + x^3 + x + 1 (0x1002B)
  - Lower 16 bits used for reduction: POLY = 0x002B

MODULES IMPLEMENTED:

1. gf_inv_16 (Sequential Inverse)
   - Algorithm: Fermat's Little Theorem, a^(-1) = a^(2^16-2) = a^65534
   - Latency: 31 cycles (15 square + 15 multiply + done)
   - Interface: clk, rst_n, start, a[15:0] → done, inv_a[15:0]

2. gf_inv_16_pipelined (Pipelined Inverse) - NEW
   - Algorithm: 15-stage pipelined square-and-multiply
   - Latency: 15 cycles initial, 1 result/cycle throughput
   - Interface: clk, rst_n, valid_in, a[15:0] → valid_out, inv_a[15:0]
   - Use case: High-throughput decoder paths

3. gf_mul_16_comb (Combinational Multiplier)
   - Algorithm: Shift-and-XOR with polynomial reduction
   - Latency: Combinational (0 cycles)
   - Interface: a[15:0], b[15:0] → p[15:0]

4. gf_inv_16_sbe_lut (Single-Bit Inverse LUT)
   - Pre-computed inverses for 2^k (k = 0..15)
   - Latency: Combinational
   - FIXED: Corrected all 16 inverse values using Fermat computation
   - Previous bug: bit 15 returned 0x002B (incorrect)
   - Corrected values:
     bit 0:  inv(0x0001) = 0x0001    bit 8:  inv(0x0100) = 0x4D0F
     bit 1:  inv(0x0002) = 0x8016    bit 9:  inv(0x0200) = 0xA793
     bit 2:  inv(0x0004) = 0x400B    bit 10: inv(0x0400) = 0xD2DD
     bit 3:  inv(0x0008) = 0xA011    bit 11: inv(0x0800) = 0xE87A
     bit 4:  inv(0x0010) = 0xD014    bit 12: inv(0x1000) = 0x743D
     bit 5:  inv(0x0020) = 0x680A    bit 13: inv(0x2000) = 0xBA0A
     bit 6:  inv(0x0040) = 0x3405    bit 14: inv(0x4000) = 0x5D05
     bit 7:  inv(0x0080) = 0x9A1E    bit 15: inv(0x8000) = 0xAF9E

5. t_value_rom (T-Value ROM for H-matrix)
   - Stores 17 T-values for RS16(19,17)
   - T[0]=0x00AC, T[1]=0x0056, T[2]=0x002B
   - T[3..16] = 0x8000, 0x4000, ..., 0x0004 (powers of 2)

6. t_sum_search (Sequential T-Sum Search)
   - Searches for (i,j) where T[i] XOR T[j] = t_sum_in
   - Latency: 1-136 cycles (worst case)
   - Use case: Weight=0 DBE detection

7. t_sum_lut_parallel (Parallel T-Sum Lookup)
   - 136 parallel comparators for all (17 choose 2) pairs
   - Latency: 1 cycle
   - Use case: Low-latency weight=0 DBE detection

12.3 ENCODER (ecc_encoder.v)
--------------------------------------------------------------------------------

RS16(19,17) ENCODER:
- Input: 272-bit data (17 × 16-bit symbols)
- Output: 304-bit codeword (17 data + 2 parity symbols)

PARITY GENERATION (H-matrix based):
  P0 = D0 ⊕ D1 ⊕ D2 ⊕ ... ⊕ D16
  P1 = T0·D0 ⊕ T1·D1 ⊕ T2·D2 ⊕ ... ⊕ T16·D16

Where Ti are the T-values from the H-matrix.

--------------------------------------------------------------------------------
12.4 DECODER (ecc_decoder.v)
--------------------------------------------------------------------------------

DBB-ECC DECODER ARCHITECTURE:
1. Syndrome Generator - computes S0, S1 from received codeword
2. SSE Locator - standard RS decoding for single symbol errors
3. DBE Locator - novel double bit error correction
   - Syndrome Splitter
   - Sublocator 1 (count=2)
   - Sublocator 2 (count=1)
   - Sublocator 3 (count=0, uses T-sum search)
4. Decision Logic - selects valid error info
5. Error Corrector - XOR operations

ERROR WEIGHT DETECTION:
  weight = popcount(S0[15:0])
  - weight > 2: Uncorrectable error (UE)
  - weight = 2: Type 1 or Type 2 DBE candidate
  - weight = 1: Type 3 DBE candidate
  - weight = 0: Type 1/2 with E_i0 = E_i1, or no error

SEVERITY ENCODING (JEDEC compliant):
  SEV[1:0] = 00: No Error (NE)
  SEV[1:0] = 01: Corrected Single-bit Error (CEs)
  SEV[1:0] = 11: Corrected Multi-bit Error (CEm)
  SEV[1:0] = 10: Uncorrectable Error (UE)
